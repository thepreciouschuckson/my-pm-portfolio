UX Research & Feature Improvement

Brassmoney: Target Savings Flexible Deduction

1. Overview & Objective

Product Context: Brassmoney is a fintech application that helps users manage finances and save towards specific goals (rent, travel, emergency funds). The Target Savings feature allows users to automate these savings based on defined rules.

Strategic Objective: Improve flexibility in the Target Savings feature by allowing users to choose their preferred deduction time, with the goal of reducing drop-offs during goal setup and increasing completion rates.

2. The Problem Space

Savings features play a key role in user retention for fintech products. If users fail to complete the setup, they never reach the core value of the feature. Currently, users are restricted to a fixed deduction time of 8:30 AM. This creates severe friction for users whose cash flow does not align with that specific time.

Current User Flow & Friction Point:

  1. User selects target savings.
  2. User sets amount and frequency.
  3. User sees fixed time (8:30 AM) and feels a lack of control.
  4. FRICTION: User exits setup without completing the goal.

Problem Statement: Fixed deduction time reduces user control. Users whose cash flow does not align with 8:30 AM fail to complete setup, leading to drop-offs and fewer completed savings goals.

3. Proposed Solution & User Stories

Introduce a time-picker during goal setup, allowing users to select their preferred deduction time.

  • As a user, I want to choose the time my savings are deducted so it aligns with when funds are available.
  • As a user, I want to edit the default time so I feel in control of my savings.
  • As a user, I want the app to remember my preferred time so I do not reset it each time.

4. Scope Management

✅ In Scope
  • UI update to savings setup screen (Time Picker).
  • Backend scheduling logic for variable times.
  • Notification updates based on the newly selected time.
🚫 Out of Scope
  • Changing overall savings frequency rules.
  • Predictive AI time recommendations based on spending habits.

5. Metrics, Risks & Next Steps

Success Metrics

  • Reduction in drop-off rate during goal setup.
  • Increase in completed savings setups.
  • Retention rate of users who create savings goals.

Trade-offs & Risks

  • Increased backend complexity due to flexible CRON scheduling.
  • Risk of failed deductions if users select poor timing.
  • Need for balance checks before execution.

Next Steps (Validation)

Before writing a line of code, I would validate this problem with user interviews, analyze the current drop-off data in Mixpanel/Amplitude, and run an A/B test comparing fixed vs. flexible timing to monitor the true impact on completion and retention.

PM Reflection: Small UX constraints can lead to significant user drop-offs. Product decisions must constantly balance user autonomy (flexibility) with system stability (backend complexity and risk of failed transactions).
